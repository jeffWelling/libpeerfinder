Background

 

When you download a bit torrent file, you download basically a file which contains a hash of the file your looking for, and a list of places to go to in order to initially connect to the swarm. The weak point in this link is the initial connection to the swarm. If someone wanted to take down the torrent, all they have to do is go for the trackers that are listed in the .torrent file.

This is mitigated in torrrents with the use of DHTs, but you still need clients to connect to initially in order to query them for the peers they know of. If you've never connected to the network before, you may not even have a list of peers from your last session to try and connect to. This is demonstrated in the Limewire shutdown

This is where libpeerfinder comes in. To continue our bit torrent example, bit torrent uses trackers and DHTs (we're pretending thats all it really uses for the purposes of this example) to locate other peers that have the same .torrent file that your looking for. So, you, or your torrent client, or a torrent client plugin even, would call libpeerfinder with the torrent command line, and it would return a list of appropriate IPs to try connecting to as trackers and or DHTs.

Without getting rid of that one last single point of failure, we don't have a truely decentralized system, and thats what libpeerfinder is intended to help with. If a method of finding that first peer is available in library form and also as a command line program, then that opens up the option of true decentralization for other developers of other peer to peer systems and protocols.

We are trying to eliminate the need for a bootstrapping node. http://en.wikipedia.org/wiki/Bootstrapping_node
